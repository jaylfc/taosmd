## 1. Abstract

taOSmd is a local-first memory system for AI agents. It stores conversation turns in a zero-loss append-only archive, layers hybrid retrieval on top, and runs offline on modest hardware without cloud dependencies.

The current LongMemEval-S headline is 97.0% Recall@5 (500 questions, full test set), a retrieval metric: the correct evidence session appears in the top-5 retrieved. This is the same metric MemPalace (96.6%) and agentmemory (95.2%) publish, so it is a like-for-like comparison taOSmd leads. An earlier edition of this report and the public docs labelled this number "end-to-end Judge accuracy"; that was a mislabel, corrected 2026-06-14 (see F-001 and the revision log). A genuine end-to-end Judge measurement (retrieve, generate, grade) with the current stack is pre-registered as E-012.

On LoCoMo (1540 QAs, 10 multi-session conversations), the current leader recipe is MaxSim + reranking: qwen3.5:9b with retrieval-top-k 50, adjacent-turns 2, bge-v2-m3 reranker, and mem0_additive fusion. Full-1540 tri-judge scores: 0.748 lenient (gemma4:e2b), 0.394 strict (llama3.1:8b), 0.659 strict (qwen3:4b-instruct-2507). All judges are external to the generator family.

A reranker-free late-interaction lever (token-level MaxSim via answerai-colbert-small-v1, backbone path) scores 0.716 / 0.388 / 0.656 on the same full-1540 tri-judge and runs in under 110ms per query on a CPU-only 16-core VPS, making it the recommended recipe for tiers where a cross-encoder download is not affordable. All numbers are under external judges with no same-family inflation.

On BEAM (mem0's long-context memory benchmark, ICLR 2026), graded by a port of mem0's own nugget-rubric judge, the full local stack scores 47.0% at the 100K tier (n=100) and 37.5% at 1M (75/200). mem0's frontier numbers are 64 to 70% at 1M, so local-9B lands honestly behind: BEAM rewards frontier generation, and the value here is an early, fully-provenanced local-tier number rather than a leaderboard win (F-014). A seven-arm config sweep found no setting that beats the 47% plateau beyond sampling noise, and showed that the CoVe self-verification that dominated on LongMemEval-S does not transfer to BEAM (N-013). These BEAM numbers carry a harness caveat surfaced 2026-06-18: the runner's 60s generation timeout can store empty answers that the nugget judge then passes on abstention rubrics, so a hardened re-confirm with an empty-answer-equals-FAIL guard was pre-registered (N-014, E-013). That re-confirm has now RESOLVED (2026-06-19): the hardened BEAM-100K re-run reproduced 49.0% with abstention at 100% and zero empty answers, so the GPU 9B numbers are audited clean, F-014 stands as measured, and N-014 is confined to slow-CPU generation. A matching hardened BEAM-1M re-run (E-017, 2026-06-20) likewise reproduced the 1M number within noise: 40.0% at n=200, abstention 100%, zero empty fires.

## 2. Methodology

### 2.1 External multi-judge protocol

The core rule is that every accuracy number uses a judge model from a different family than the generator. Same-model judging is measured to inflate reported scores by 15 to 22 points on the LoCoMo task (source: docs/benchmarks.md "Judge sensitivity" section). On our own stack, the qwen3.5:9b generator self-judging its own output read 0.64 overall vs the external judge reading of 0.54 on the same predictions, a +0.10 inflation. That is consistent with the historic estimate in direction, somewhat below it in magnitude.

The tri-judge panel used for all full-1540 measurements (Jun 2026):

- **Lenient judge: gemma4:e2b.** The most lenient open-source judge we measured. Calibrated closer to gpt-4o-mini behavior: accepts inferred answers, is forgiving on minor wording differences. Useful for comparing against published SOTA numbers that also use a lenient API judge.
- **Strict judge 1: llama3.1:8b.** A non-thinking model that emits clean YES/NO verdicts. Compresses the field: it grades very harshly and the strict judge penalizes any answer that isn't nearly verbatim with the reference.
- **Strict judge 2: qwen3:4b-instruct-2507.** The non-thinking HF GGUF build of qwen3:4b-instruct. Used because the thinking-enabled qwen3:4b Ollama build no longer emits clean YES/NO verdicts in its current version and was retired as a judge in Jun 2026. Legacy leaderboard figures (for example RRF 0.557, mem0 0.540) came from the older thinking-mode qwen3:4b and are kept in docs/benchmarks.md for history but are not directly comparable to new strict-instruct numbers.

A fourth column, the **community-standard judge qwen3:14b**, is used to make late-interaction numbers comparable beyond the project's own panel. qwen3:14b is the official judge of the LoCoMo-Refined benchmark.

The legacy single-judge leaderboard in docs/benchmarks.md (entries from before Jun 2026) used the thinking-mode qwen3:4b and is kept for history. New cells added from Jun 2026 onward carry all three tri-judge scores.

For retrieval-only measurements the team uses **judge-free R@K**: the share of questions whose annotated evidence lands in the retrieved set, measured by the retrieval-latency probe without any LLM generation or grading. R@K is objective and does not depend on judge calibration.

### 2.2 Datasets

- **LongMemEval-S.** 500 questions, standard test set. Single and multi-session. Benchmark for end-to-end answer quality with the full pipeline. Dataset: github.com/xiaowu0162/LongMemEval (ICLR 2025).
- **LoCoMo-10.** 10 multi-session conversations, 1540 question-answer pairs across four categories: Single-hop, Temporal, Multi-hop, Open-domain. Conversations span 50+ sessions and 400-700 turns. A harder evaluation than LongMemEval-S: more categories, more QAs, and longer conversations. Benchmark harness: benchmarks/locomo_runner.py. Source: snap-research LoCoMo dataset.
- **LoCoMo-Refined (subset-comparable).** The revised 1382-question set from the LoCoMo-Refined leaderboard, using the corrected answer key and the official qwen3:14b judge. A full run was attempted but the official qwen3:14b judge proved unreliable under local Ollama (see E-003), so no refined-set number is reported. Our qwen3:14b column is on the original 1540-question set; it is judge-comparable to the Refined leaderboard but not set-comparable.
- **BEAM (100K / 1M / 10M).** mem0's long-context memory benchmark (ICLR 2026): conversations of 100K to 10M tokens with 2000+ nugget-graded questions across ten ability categories. Graded by mem0's own nugget-rubric judge (an answer passes at score at or above 0.5), ported in benchmarks/beam_runner.py so the grading matches mem0's published methodology. The dataset is public and ungated on HuggingFace (Mohammadta/BEAM for the 100K-1M tiers, Mohammadta/BEAM-10M). Used for an honest local-tier head-to-head against mem0's frontier numbers (1M 70.1% at top-200, 10M 50.5%).

### 2.3 Metrics

**End-to-end Judge accuracy** is the primary accuracy metric: what fraction of generated answers get marked correct by the judge, graded against the gold reference. It combines retrieval, generation, and grading in one number.

**R@K** (Retrieval at K) is used for retrieval-only levers where there is no LLM generation: what share of questions have their annotated evidence passage in the top-K retrieved results. It is judge-free and directly reflects retrieval quality.

**Latency** is reported as p50 and p95 per-query wall-clock in milliseconds. **Context tokens** are estimated from context chars / 4 in the LoCoMo runner and carried in the result JSON per row.

Per the three-number reporting rule: every accuracy claim in this report is accompanied by latency and context cost where those have been measured.

### 2.4 Hardware tiers

- **Orange Pi 5 Plus, 16 GB RAM (RK3588 NPU).** Reference low-end tier. The LongMemEval-S 97.0% Recall@5 retrieval stack was measured here (embedder: all-MiniLM-L6-v2 ONNX on CPU; Recall@5 is retrieval-only so no generator is involved in that number). The tier's generator for judged LoCoMo work is qwen3-4B via rkllama on the NPU. Source: docs/benchmarks.md hardware tiers section.
- **16-core x86 CPU VPS (no GPU).** Used for the retrieval-latency probe only. No LLM inference; embedder runs on CPU with ONNX Runtime. Source: docs/benchmarks.md "CPU-tier viability" subsection.
- **Fedora host, RTX 3060 12 GB.** Primary LoCoMo benchmark host. All full-1540 tri-judge numbers were measured here. Generator: qwen3.5:9b Q4_K_M via Ollama. Source: docs/benchmarks.md LoCoMo section.
- **GTX 1050 Ti LXC, 4 GB VRAM.** Measured at 0.530 ext judge (qwen3:4b, LXC container with CUDA). Source: docs/benchmarks.md "4 GB GPU" hardware tier.

Experiments enter section 6 with a declared design and kill criterion before results exist. When results land, the experiment moves to section 3 or 4 with the original kill criterion quoted verbatim. Editing a kill criterion after results exist is falsification and is not done.

### 3.5 Extraction-Hallucination Rate (F-009)

A cross-family claim-verification pass measured how often the extraction step stores claims that are not fully supported by their source turns. gemma4:e2b extracted claims; a different family, qwen3:4b-instruct-2507, judged each claim SUPPORTED, PARTIAL, or UNSUPPORTED against the exact source text the extractor saw. Using different families on the two sides guards against a model rubber-stamping its own output.

Across 526 claims from three conversations, the PARTIAL plus UNSUPPORTED rate was 18.8 percent (verifier error rate 0.2 percent, one claim). Per conversation: conv-26 22.0 percent (168 claims), conv-30 12.6 percent (103 claims), conv-41 19.2 percent (255 claims). In plain terms, close to one in five extracted facts is not fully supported by the turn it was drawn from. This is a measurement nobody else in this space reports, and it is the quantitative case for the provenance-and-verification direction: a memory layer that keeps the source can measure and re-verify its own claims, while an extraction-only layer cannot.

Source: bench host results 20260611 e2 claim-verifier pairs and summary.

### 3.8 BEAM long-context (mem0 nugget-rubric judge)

BEAM (mem0, ICLR 2026) is a long-context memory benchmark at the 100K, 1M, and 10M token scales, graded by a nugget-rubric judge where an answer passes at score 0.5 or above. It is the head-to-head where mem0 publishes frontier numbers (1M 70.1% at top-200, 10M 50.5%), so it is the cleanest place to state an honest local-tier number against a public frontier baseline. taOSmd runs its own port of the BEAM nugget judge (benchmarks/beam_runner.py) so the grading matches mem0's methodology, over the full local stack: arctic-embed-s, tight retrieval (6 chunks, FTS 3, bge-v2-m3 rerank to top 4, 2000 assemble tokens, 6000 context chars), qwen3.5:9b generator, Qwen3-4B-Instruct judge.

Measured (F-014): BEAM-100K 47.0% pass (n=100), BEAM-1M 37.5% (75/200). Against mem0's 64 to 70% at 1M, local-9B lands honestly behind, exactly as the thesis predicts. BEAM rewards frontier generation, so the value is the early, fully-provenanced local-tier number, not a leaderboard win. The intelligent-context-release lever from the earlier context-amount sweep is already in this configuration: tight 6-chunk context beats a rich 40-chunk context by +16pp on the 9B, because more context buries a small model.

A seven-arm config sweep on BEAM-100K (chunk size, chunking mode, self-verify, decomposition; n=100 per arm) found no lever that beats the 47% plateau beyond sampling noise (N-013):

| Arm (BEAM-100K, n=100) | pass% | avg_score |
|---|---|---|
| base (chunk 100, self-verify) | 47.0 | 0.436 |
| chunk 50 | 47.0 | 0.417 |
| chunk 200 | 49.0 | 0.425 |
| chunk 300 | 44.0 | 0.394 |
| sentence-mode (120 words) | 47.0 | 0.425 |
| chunk 100, self-verify OFF | 49.0 | 0.455 |
| chunk 100, decompose ON | 47.0 | 0.436 |

At n=100 the pass-rate standard error is about 5 points, so the 44 to 49 spread is one band around 47. The one signal worth carrying forward is that self-verify gives no BEAM benefit: the self-verify-off arm ties the top pass-rate and leads on avg_score (0.455). On LongMemEval-S the same CoVe self-verification was the dominant +17.8pp lever (F-013); it does not transfer to BEAM because BEAM failures are wrong answers, not abstentions, and a verify-or-abstain pass can only recover the latter. Provenance: benchmarks/results/beam_campaign_resume_20260616_221030.log (arms 3 to 7) and beam_campaign_20260616_115113.log (arms 1 to 2), branch feat/beam-runner.

A follow-up retrieval-recall sweep tested a different hypothesis: is the plateau a retrieval-recall ceiling? The tight config retrieves only about 6 dense plus 3 lexical candidates, so an exact-fact needle could be missing before generation ever sees it. Widening retrieval regressed instead of helping (all n=100, self-verify off): tight anchor 49.0%, recall_wide (40 dense + 15 lexical) 43.0%, recall_wider (60 + 20) 46.0%, fts_heavy (lexical-dominant) 41.0%. So the plateau is not a retrieval-recall gap; wider and lexical-heavy retrieval add distractors the 9B cannot filter, and tight retrieval is optimal. Two independent sweeps (config and recall) now confirm 47 to 49% as the local-9B BEAM-100K ceiling. Provenance: benchmarks/results/beam_recall_20260617_130445.log.

Harness caveat (N-014), surfaced 2026-06-18. A separate attempt to run a frontier-tier generator, qwen3:30b-a3b (30B total, about 3B active), on the idle 16-core CPU VPS exposed a measurement trap in the runner that qualifies every BEAM number above. The runner's LLM client has a hard 60s timeout (beam_runner.py). On CPU the 30B generates at roughly 15 tokens per second, so a long answer overruns 60s and the runner stores an empty string. The ported mem0 nugget judge then scores that empty answer as PASS on negative and abstention rubrics, because a non-responsive answer satisfies a "the response must not assert X" criterion (the judge prompt scores an empty or refusing response as full compliance for a negative nugget, beam_runner.py judge prompt). The effect is large: the 30B's first CPU pass read 55.0% (11/20) with 10 of 20 answers empty, and the honest re-run at a 240s timeout, with all 20 questions fully answered and zero empties, read 20.0% (4/20, avg_score 0.177, n=20). Verified on identical questions: answer length 0 in the first run versus about 1300 characters in the second, with abstention, knowledge-update, and temporal-reasoning scores flipping from pass to fail once real answers replaced the blanks. So a frontier-tier model on capable-but-CPU hardware did not beat the local 9B here; on a clean fully-answered basis the 30B scored 20% (n=20, directional, one conversation), which fits the broader finding that model quality and generation speed, not parameter count, drive the local-tier BEAM number. The 9B figures in F-014 were measured on GPU with errors=0 and fast generation, so timeout-empties are unlikely there, but the campaign logs do not record per-answer lengths and the 9B abstention category sits at 100%, the category most exposed to this artifact, so the 9B numbers cannot be cleared of it from existing files alone. A hardened re-confirm (raised timeout plus an empty-answer-equals-FAIL guard) is pre-registered as E-013, with the kill criterion in section 6. RESOLVED 2026-06-19: the hardened BEAM-100K re-run (guard plus raised timeout, feat/beam-runner f7dc94e) reproduced 49.0% pass, avg_score 0.455, errors=0, with abstention at 100% and ZERO empty-answer guard fires across all 100 questions, so F-014 stands as measured and N-014 is confined to slow-CPU generation; the 100% abstention is genuine model behaviour, not a blank-answer artifact (there were no blanks). Provenance: VPS 30B CPU run task report 2026-06-18 plus bench host benchmarks/results/beam_e013_100k_20260619_110029.log, runner beam_runner.py on branch feat/beam-runner. The matching hardened BEAM-1M re-run (E-017, section 6) followed on 2026-06-20 with the same guards and config and came back 40.0% (80/200), avg_score 0.373, errors=0, abstention 100%, zero empty-guard fires, so the 1M number also stands within noise of the original 37.5% (+2.5pp, inside the 3.5pp standard error). Provenance: bench host benchmarks/results/beam_1M_1781911762.json and e017_beam1m_20260619_222343.log, committed feat/beam-runner ca95fa3.

A generator-tier probe then asked the natural follow-up: would a bigger generator (which more capable hardware could run) lift the number? On the 12GB card, every model that fits underperformed the well-tuned 9B: gemma4:12b scored 46.0% and qwen3:14b 46.0% (both n=100, tight retrieval, same Qwen3-4B-Instruct judge), versus the 9B's 49.0%. So at the local tier the lever is generator quality, not size: the well-tuned qwen3.5:9b is the sweet spot and a bigger generic model can slightly regress (consistent with N-007, where gemma4:12b also trailed the 9B on LoCoMo). The qwen3:14b run used a new two-phase generate-then-judge runner mode (generate with the big model alone, unload it, then judge with the small judge) so the 14B fits 12GB without the judge co-resident. A genuinely frontier-class generator (32B or 70B) needs more than 12GB of VRAM to benchmark and is the open follow-up on capable hardware. Provenance: bench host /tmp/beam_gemma12b.out and /tmp/beam_14b.out, predictions benchmarks/results/beam_pred_qwen3-14b_20260617_200317.json; the two-phase mode is on branch feat/beam-runner (beam_runner.py --phase, beam_twophase.sh).

## 4. Negative Results

Negative results are recorded with the same rigor as wins. A lever that failed is a finding.

**N-001. CLAG cluster pre-filter.**

A frontier idea aimed at small-model latency: cosine k-means over the candidate pool, keeping only the query's nearest cluster(s) before ranking. On the 200-QA subset every variant regressed. The worst variant (keep nearest 1 cluster) scored -0.285 gemma / -0.205 qwen3:4b vs baseline. The keep-sweep was monotonic; relaxing the pruning moved toward baseline but never beat it. Cluster pruning only ever discards relevant evidence. The failure pattern matched HyDE, Chain-of-Memory, and thinking-mode: a lever tuned for frontier models that regresses at the compute tier targeted by taOSmd. Not shipped. A benchmark-only --clag flag is retained for reproduction.

Source: docs/benchmarks.md, CLAG cluster pre-filter section.

**N-002. Chain-of-Memory.**

Tested May 31 2026. Result: -0.16 single-hop for +0.02 overall at 1.8x latency. The paper is frontier-tuned and lossy at the compute tier this project targets. Not shipped.

Source: project memory reference_memory_landscape_may2026.md, entry: "Chain-of-Memory TESTED May 31 REGRESSED on our stack."

**N-003. Few-shot prompting.**

Adding 5 exemplars to the answer prompt on top of the leader recipe: full-stack + few-shot lands at 0.540 vs the 0.557 leader (qwen3:4b strict, full-1540). Delta: -0.017. **N-004. HyDE (Hypothetical Document Embeddings).**

Standalone HyDE scores 0.456 vs the 0.516 adj=2 baseline (qwen3:4b strict, full LoCoMo 1540), a -0.060 regression. When stacked on the leader recipe, HyDE pulls the leader from 0.557 down to 0.486: -0.071 vs the leader. Classified as a poison-pill: HyDE assumes the generator can imagine a good hypothetical answer. Small generators hallucinate the answer's structure, the embedding lookup matches that hallucination, and recall collapses. Direction-consistent at the Pi NPU tier (-0.057 vs Pi baseline). Do not enable HyDE on memory-recall benchmarks at this generator tier.

**N-008. ColBERT projected spaces do not beat the backbone at our tier (E-004 resolved).**

**N-009. Surprise-boundary chunking is a coverage artifact (E-001 resolved, both arms dead).**

The surprisal pillar's retrieval-side claims are both negative. The prior arm (weighting candidate scores by turn surprisal) was flat at +0.0000 across every weight tested. The chunking arm (merging predictable turns into chunks at surprise boundaries) showed +0.1263 at equal k, but a matched-turn-budget control kills it: baseline handed the same raw turn count directly (k=120, 0.9192) beats chunking at k=20 (0.7677, roughly 120 turns via 6-turn chunks) by 0.15. The apparent gain was retrieved-turn coverage, not surprisal signal. Methods lesson worth keeping: R@K favors any variant that packs more turns per retrieved item, so chunking-style levers must always be compared at matched turn budget, not matched k. What survives of the surprisal idea: the write-skip floor (E-006, running) and surprisal-seeded trace strength for consolidation priority (a v2 question, not a retrieval lever).

Source: VPS benchmarks/results/e1_mb_k20_20260612_083159.json and e1_mb_k120_20260612_083159.json, git 1427f54.

**N-010. Write-skip floor: safe but fires too rarely to ship (E-006 resolved).**

Skipping highly predictable short turns at ingest (surprisal z at or below -1.0 and under 12 tokens) left R@K within noise (+0.0051) and skipped zero evidence turns, so the classifier causes no measurable recall harm at these thresholds. But it only fired on 18 of 788 turns (2.3 percent), under the pre-registered 5 percent shipping floor: on LoCoMo there are simply not enough bare greetings and acknowledgements for the floor to save meaningful index weight. Not shipped. The thresholds were deliberately conservative; loosening them to hit 5 percent would change the safety result and would need a fresh pre-registration. Worth revisiting only on a corpus with denser conversational filler.

Source: VPS benchmarks/results/e1_write_skip_20260612_143625.json, variant code on branch bench/e1-write-skip (46793e0).

**N-011. Surprisal is a bad retention-priority signal (the v2 consolidation kill-shot fails).**

The last surviving role for the v2 surprisal pillar was consolidation priority: when an agent must forget down to a budget, is surprisal a good "keep" signal? It is the worst one tested. Under retention budgets of 25/50/75 percent on subset-200 (judge-free R@K, evidence on a dropped turn counts as a miss), keeping the highest-surprisal turns scored 0.313/0.444/0.591, below random (0.258/0.424/0.571) at every budget, and far below the winner. The winner is length: keep the longest turns, 0.525/0.641/0.702. Recency was weakest. The reading: high token-surprisal marks unpredictable content, not evidence-bearing content, and short surprising turns (the ones the write-skip floor also targeted) carry little retrievable evidence, so prioritising them throws away the longer, denser turns that answers actually need. With its retrieval role already dead (N-009, N-010), the surprisal pillar of v2 is conclusively killed; the spine pivots to the provenance and claims layer (F-009). The length result is a genuine positive side-finding, a cheap, model-free retention heuristic, recorded for its own future pre-registration rather than claimed here.

Source: VPS benchmarks/results/e008_retention_20260613_115258.json, branch bench/e1-retention.

**N-013. BEAM-100K config sweep: no lever beats the 47% plateau; self-verify is BEAM-neutral.**

Seven arms (chunk size 50/100/200/300, sentence-mode, self-verify on and off, decomposition on) on BEAM-100K at n=100 each, over the full local stack (arctic-embed-s, tight retrieval, qwen3.5:9b generator, Qwen3-4B-Instruct judge). Every arm landed in the 44 to 49% band, one standard error around the 47% tight-retrieval baseline. chunk200 and self-verify-off are nominally highest at 49% but within noise, so no configuration lever moves the local-9B BEAM-100K number. The transferable sub-finding: CoVe answer self-verification, the dominant +17.8pp lever on LongMemEval-S (F-013), gives no BEAM benefit; the self-verify-off arm ties the top pass-rate and leads avg_score (0.455), because BEAM failures are wrong answers rather than abstentions and a verify-or-abstain pass can only recover abstentions. The 47% plateau confirms local-9B sits structurally behind frontier generation on BEAM (mem0 64 to 70% at 1M); the product story is the provenanced local-tier number, not a tuned score. The full per-arm table is in section 3.8.

**N-014. BEAM harness empty-answer confound: a 60s LLM timeout plus a judge that passes empties inflates the score.**

This is a methodology negative, not a lever, and it qualifies every BEAM number in this report. The runner's LLM client has a hard 60s timeout. When generation overruns it (the case on slow CPU hardware), the runner stores an empty answer rather than failing the question. The ported mem0 nugget judge then scores an empty or non-responsive answer as full compliance on negative and abstention rubrics, so blanks pass. Caught while running qwen3:30b-a3b on the CPU-only VPS: the first pass read 55.0% (11/20) with 10 of 20 answers empty, and the honest re-run at a 240s timeout with zero empties read 20.0% (4/20, avg_score 0.177, n=20), confirmed by comparing answer lengths (0 versus about 1300 characters) on identical questions with abstention, knowledge-update, and temporal scores flipping from pass to fail. Two consequences. First, a frontier-tier model on CPU did not beat the tuned local 9B (20% versus 49%, though not yet a like-for-like comparison), reinforcing that quality and speed beat parameter count at the local tier. Second, the 9B figures in F-014 were measured on GPU with errors=0 and fast generation, so they are probably sound, but they cannot be cleared of this artifact from the existing logs (no per-answer lengths recorded, and abstention sits at 100%), so a hardened re-confirm is pre-registered (E-013). The fix the re-confirm builds and ships: treat an empty or whitespace-only answer as an automatic FAIL before the judge runs, and raise the timeout so generation never truncates to empty. Provenance: VPS 30B CPU run task report 2026-06-18, runner beam_runner.py on branch feat/beam-runner.

Source: benchmarks/results/beam_campaign_resume_20260616_221030.log and beam_campaign_20260616_115113.log, branch feat/beam-runner. A follow-up retrieval-recall sweep (anchor 49.0 / recall_wide 43.0 / recall_wider 46.0 / fts_heavy 41.0, n=100, self-verify off) confirms tight retrieval is optimal and the plateau is not a retrieval-recall ceiling: wider and lexical-heavy retrieval regress. Provenance benchmarks/results/beam_recall_20260617_130445.log.

**N-015. MemX R1 reject gate is inert at the default tau on LoCoMo with arctic-embed-s.**

The MemX low-confidence reject gate (rule R1) abstains before generation when the lexical recall set is empty AND the maximum dense cosine similarity is below tau (default 0.50). We built it as an opt-in default-off lever and ran the pre-registered E-014 ablation, judge-free, on LoCoMo (200 QAs, strategy full, arctic-embed-s, gate off versus on). R@K is identical: 0.73 off and 0.73 on, with F1 (0.20) and BLEU (0.13) also byte-identical across both arms. The gate never fired, because arctic-embed-s produces cosine similarities that broadly clear 0.50, so the abstention condition never triggers at the default tau. Against the kill criterion (ship as opt-in only if R@K drop is at most 0.02), the 0.00 drop formally clears the floor, but the pass is hollow: the gate is a no-op at the default tau on this embedder, neither costing recall nor adding value. The honest conclusion is that the gate is safe but provides no measurable benefit on LoCoMo, whose questions are answerable; its tau must be calibrated to the embedder's similarity distribution to do anything, and its real value (abstaining on genuinely unanswerable or low-confidence queries) needs a benchmark with such queries, which LoCoMo is not (the same applicability limit as E-005). The gate stays default-off as a documented opt-in. Provenance: bench host benchmarks/results/e014_gate0_v3.log and e014_gate1_v3.log, feat/memx-levers 96b7725.

**N-017. The LongMemEval end-to-end Judge parser scored every INCORRECT verdict as a pass; all `--llm` Judge numbers since 2026-04-13 were inflated.**

This is a methodology negative, not a lever, and it qualifies every LongMemEval-S end-to-end Judge number in this report (F-012, F-013, E-021, E-022). The runner's judge function (benchmarks/longmemeval_runner.py, score_answer_llm) prompted the judge for one word, CORRECT or INCORRECT, uppercased the reply, and scored it with `"CORRECT" in judgment`. The string "INCORRECT" contains the substring "CORRECT", so every INCORRECT verdict was counted as a pass. The only answers scored wrong were ones a separate abstain guard caught first, so the Judge score measured whether the model attempted an answer, not whether the answer was right, which is why better retrieval appeared to raise it. The bug was present from the first commit of the runner (2026-04-13) and was caught 2026-06-23 during a cross-family judge firm-up: re-judging the same saved predictions with a corrected parser dropped an n=100 sample from a stored 77/100 to 33/100. Fixed the same day in PR #176 (a `_parse_verdict` helper that checks INCORRECT first and fails closed, with 5 regression tests), merged to master.

Corrected numbers, same fixed Qwen3-4B-Instruct judge. The full-500 arctic re-baseline reads 214/500 (42.8%) on the Qwen judge and 256/500 (51.2%) under a cross-family llama3.1:8b re-judge, so the real end-to-end Judge sits in a 43 to 51 percent band depending on judge severity, not the 74.6 percent of F-013. The empty-generation inflation that tripped E-019 is absent here (0/100 empties, none scored correct), and the same-family Qwen judge, once the parser is fixed, is the stricter of the two.

A generator finding follows from the fix. A bake-off (n=150 representative, seed 7, all six question types, judge held constant, only the generator varied) put gemma4:12b at 53.3 percent versus the shipped qwen3.5:9b at 43.3 percent, a +10pp win on less VRAM at about 1.6x latency; qwen3:14b is +6pp; phi-4-reasoning:14b offloads on the 12GB tier and no qwen3.6 dense variant fits it. Re-running the local-first head-to-head with gemma4:12b driving both arms flips the prior dead heat: taOSmd-combo 57.3 Qwen / 64.7 llama versus MemOS-lossless 52.7 Qwen / 64.0 llama, so on n=150 taOSmd is ahead on the strict judge (+4.6) and even on the lenient one (+0.7), where the old generator had MemOS ahead by 6 on the strict judge. The pre-registered full-500 confirm (2026-06-23) then lands it at PARITY: combo 53.8% Qwen / 61.4% llama versus MemOS-lossless 54.0% / 61.2%, a statistical tie on both judges. So the n=150 +4.6 strict lead was sampling noise; the honest verdict is that with the gemma4:12b generator taOSmd-combo MATCHES MemOS-lossless, it does not beat it. The generator upgrade is real on LongMemEval (gemma4 full-500 combo 53.8 Qwen vs the shipped qwen3.5:9b 42.8), but it is TASK-DEPENDENT and does not transfer: gemma4:12b UNDERperforms qwen3.5:9b on LoCoMo (0.63 vs 0.68, N-007) and BEAM (46 vs 49, rev 1.25). So gemma4:12b wins only the single-fact retrieval-QA slice; the shipped default generator STAYS qwen3.5:9b (the multi-session and long-context leader), and gemma4:12b is a documented per-task option, not a global default flip.

Scope. The bug is isolated to benchmarks/longmemeval_runner.py (grep-confirmed, no other runner shares the pattern). The Recall@5 headline (F-001), the LoCoMo numbers (separate locomo_runner judge), the BEAM numbers (separate nugget-rubric judge), and the served-hallucination metrics are not affected. Provenance: PR #176 (master); bench logs e022_rebaseline_arctic500_20260623_042130.log, benchmarks/results/e022_bakeoff/, and e022_combo_gemma12b / e022_memos_gemma12b _20260623 with their .rescored_llama3_1_8b.json.

## 6. Ongoing Work (Pre-registered)

These experiments have declared designs and kill criteria. Results will be recorded verbatim whatever they are.

**E-001. Surprisal retrieval prior and surprise-boundary chunking.**

Hypothesis: turns that carry high surprisal relative to the running conversation are more likely to contain important facts worth preferential retrieval. The lever would weight candidate scoring by a surprisal signal estimated from the embedding shift between turns, and use high-surprisal points as chunk boundaries.

Kill criterion (verbatim): "no variant beats baseline R@K by more than 0.02 on subset-200."

Result (2026-06-11), recorded honestly: the surprisal PRIOR variant was flat, +0.0000 R@K at every weight (0.25, 0.5, 1.0) over a 0.641 baseline on subset-200. The prior path is dead. The surprise-boundary CHUNKING variant reported +0.0859 (0.727), which clears the kill threshold on paper, BUT the result is confounded and is NOT being claimed as a win: the chunker produced a mean chunk length of 22.54 turns against a stated maximum of 6, which means its size cap was not being enforced, and evidence was credited per-chunk rather than per-turn, so a single giant chunk trivially counted as a hit for every turn it spanned. **E-005. Temporal date-range lever.**

Origin, stated honestly: this is a port of the temporal parser from an earlier internal project (engram), which scored 94.6 percent on LoCoMo temporal in one configuration. Full scan: single-hop 2.1 percent, temporal 9.3 percent, multi-hop 2.1 percent, open-domain 15.8 percent, overall 171/1540 (11.1 percent). **E-008. Surprisal as a forgetting/retention-priority signal under a memory budget.**

The consolidation kill-shot for the v2 surprisal pillar, designed to test a claim today's negatives do NOT touch. N-009 and N-010 tested surprisal at retrieval time over the full corpus (which turns to chunk, skip, or boost when answering). E-008 tests it as the offline retention signal, the core sleep-consolidation decision: when an agent must forget down to a working-set budget B (the archive keeps everything; this is only what stays in the live compilation), is surprisal a better "keep" signal than the cheap baselines?

Design: per conversation, score per-turn surprisal (reuse the E1 probe's cached scores). This is distinct from write-skip (N-010): that was a fixed low-surprisal-short threshold firing on 2.3 percent of turns; this is a retention curve where surprisal selects across all turns at several budgets against recency, the baseline it must beat to justify the scorer's cost.

Kill criterion (verbatim): "surprisal-priority retention must beat the best non-random baseline (recency or length) by more than 0.02 R@K at one or more matched budgets and must not trail it at any budget, for the consolidation-strength bet to survive. Result (2026-06-13), KILLED, recorded as N-011. Judge-free R@K under retention budgets on subset-200, keep-by-policy vs the baselines:

|---|---|---|---|---|---|
| 25 percent | 0.313 | 0.232 | 0.525 | 0.258 | -0.212 |
| 50 percent | 0.444 | 0.348 | 0.641 | 0.424 | -0.197 |
| 75 percent | 0.591 | 0.450 | 0.702 | 0.571 | -0.111 |

Quoting the criterion: "surprisal-priority retention must beat the best non-random baseline by more than 0.02 at >=1 budget and must not trail it at any budget. Surprisal not only misses, it is the WORST signal at every budget, below random: keeping the highest-surprisal turns is actively bad for what to retain. ### 6.x E-014: MemX R1 low-confidence reject gate on LoCoMo retrieval

Pre-registered 2026-06-19, before any ablation run. Motivation: the MemX paper (arXiv:2603.16171) proposes a low-confidence rejection gate (rule R1) that abstains before generation when retrieval confidence is low, to avoid answering from weak evidence. We built it as an opt-in, default-off lever (feat/memx-levers 96b7725, taosmd/retrieval.py apply_reject_gate, env-gated TAOSMD_REJECT_GATE and TAOSMD_REJECT_TAU). The gate abstains (returns an empty result set) when BOTH the lexical recall set is empty AND the maximum dense cosine similarity is below tau (default 0.50). This experiment measures its recall cost on LoCoMo.

Design. LoCoMo (200 QAs across 10 conversations, arctic-embed-s onnx embedder, strategy full), with TAOSMD_REJECT_GATE off (baseline) versus on (tau 0.50), judge-free (no inline judge). Primary metric: R@K (retrieval_recall, the share of questions whose evidence is retrieved), which is objective and judge-independent. Secondary: the gate firing rate (how many questions it abstained on). Scope note: the locomo_runner full strategy wires only the dense (vector) source, so the gate's lexical-rescue branch never fires here; this measures the MOST aggressive form of the gate, pure dense-confidence abstention. The full gate, with an FTS source present, would abstain strictly less.

KILL CRITERION (written before results): the reject gate qualifies to ship as an opt-in low-confidence mode only if it drops R@K by no more than 0.02 (the same shipping-floor threshold used for E-006 and E-009). If R@K drops by more than 0.02, the gate is too aggressive for the LoCoMo regime, where questions are overwhelmingly answerable, so it abstains on answerable queries and loses recall; that is recorded as a negative result (default-off, with the measured recall cost and firing rate documented), and the gate's natural habitat is noted as open-domain or unanswerable queries that LoCoMo underrepresents (cf. E-005, where LoCoMo could not fairly measure the temporal lever). Either outcome is recorded.

Resolved 2026-06-19. The judge-free R@K ablation ran on LoCoMo (200 QAs, 10 conversations, arctic-embed-s, strategy full, gate off versus on at tau 0.50). Result: R@K is IDENTICAL, 0.73 with the gate off and 0.73 with the gate on, and F1 (0.20) and BLEU (0.13) are byte-identical across both arms. The gate never fired. The reason is calibration: arctic-embed-s produces cosine similarities that broadly exceed 0.50, so the abstention condition (no lexical hit AND max dense similarity below tau) never triggers at the default tau. Quoting the criterion: "the reject gate qualifies to ship as an opt-in low-confidence mode only if it drops R@K by no more than 0.02." The R@K drop is 0.00, so it formally clears the floor, but the pass is hollow: the gate is inert at the default tau on this embedder, neither costing recall nor providing any benefit. Honest reading: the gate is SAFE (zero recall cost at the shipped default) but provides NO measurable benefit on LoCoMo, because LoCoMo questions are answerable and arctic's similarities clear the threshold. Its tau must be calibrated to the embedder's similarity distribution to do anything, and its actual value, abstaining on genuinely unanswerable or low-confidence queries, cannot be measured on answerable LoCoMo (the same limitation as E-005, where LoCoMo could not fairly test the temporal lever). Recorded as N-015. The gate stays default-off, shipped as a documented opt-in with a note to calibrate tau per embedder; the natural follow-up if wanted is a tau sweep to map where it starts firing and the recall-cost curve. Provenance: bench host benchmarks/results/e014_gate0_v3.log and e014_gate1_v3.log, feat/memx-levers 96b7725.

Status: resolved to N-015 (reject gate inert at default tau 0.50 on LoCoMo plus arctic; R@K 0.73 unchanged, never fires; safe but no benefit; ships default-off opt-in, tau needs per-embedder calibration).

### 6.x E-019: Independent third-party harness (maximem-ai eval harness)

Pre-registered 2026-06-21, run queued for the next bench-GPU window. Motivation: every accuracy number published so far comes from our own harnesses (our LongMemEval-S, LoCoMo, BEAM, and EventQA runners). An independent third-party harness we did not author is a stronger integrity signal, and it places the local-tier stack against other memory systems on a surface they all share.

Design. The maximem-ai/memory_and_context_eval_harness (MIT-licensed, public, commit d811a5f) evaluates LoCoMo (1540 questions) and LongMemEval-S (500 questions). It exposes a Provider ABC (ingest / search / clear) and generates the answer itself from the provider's retrieved context, then grades with an LLM judge; both the answer model and the judge model are reached through the OpenAI SDK and honour OPENAI_BASE_URL, so the whole pipeline runs against a local Ollama with zero external API calls. The local stack is wired in with a Provider adapter over taosmd serve (text maps to the harness `memory` field, confidence to `score`, source and timestamp and metadata to `source_evidence`; adapter staged at /tmp/maximem-harness/adapters/taosmd_adapter.py). Run config: taosmd serve with the recommended retrieve recipe plus arctic-embed-s, the answer model qwen3.5:9b and the judge model from a DIFFERENT family (llama3.1:8b), the harness's own default judge prompts, the real downloaded datasets (not the sample fallback), a 20-question subset first, then the full set.

KILL CRITERION (validity, fixed before the run). This is a measurement experiment, not a lever test: an honest local-tier number is the finding whether it is high or low, exactly as with BEAM (F-014). So the criterion governs VALIDITY, not a score threshold. ### 6.x E-020: Strict re-judge of the maximem LoCoMo predictions

### 6.x E-021: Generator-prompt ablation (can a better answer prompt beat 74.6%)

Pre-registered 2026-06-22, queued overnight behind E-020. Design. A representative n=100 screen (all six question types, fixed seed), then a full-500 confirm of any arm that clears the screen.

KILL CRITERION (fixed before the run). An arm SHIPS only if it beats the baseline by more than the sampling noise (n=100 screen, expect about +/-5pp; full-500 confirm reported with its SE) under the fixed strict judge AND the gain is reproduced by the cross-family llama3.1:8b firm-up (judge-robust, the F-013 standard). Arms that do not clear are recorded as negatives with the same rigor as a win; the pre-registered priors are that the persona arm is a dud and self-consistency is uncertain (it helps verifiable-majority tasks more than open-ended recall, and self-correction is known to help free generation while hurting multiple-choice, consistent with our own N-013). ### 6.x E-022: Local-first head-to-head scoreboard (where does taOSmd place)

Pre-registered 2026-06-22 after a recon of the two strongest local-first peers. Motivation: "local-first offline memory" is no longer open ground (a June 2026 scan found MemOS, OpenMemory, mem0's local mode, Memory Kit, MemX and others), and there is no shared leaderboard, so any ranking is a judgment across incommensurable systems with mostly unprovenanced numbers. The honest way to answer "where do we place" is a like-for-like number: run the peers through OUR harness with everything held constant except the memory backend.

Design. One harness, three retrieval backends: taOSmd, MemOS (MemTensor/MemOS, Apache-2.0), and OpenMemory (CaviraOSS/OpenMemory, Apache-2.0). The same data (LongMemEval-S oracle, the 500-question set where F-013 was measured), the SAME local generator (qwen3.5:9b), and the SAME frozen strict Qwen3-4B-Instruct judge across all three; only the memory backend varies (ingest the sessions, retrieve top-k per question, feed that context to the fixed generator + judge). All three run fully local with a local Ollama embedder and an on-disk store, retrieve-only (none is allowed to generate its own answer, so the generator and judge are constant). Per the recon, each system is run in TWO configs where they differ: the FAIR lossless-retrieval config (MemOS general_text raw-store with no write/read LLM and no decay; OpenMemory via the Python SDK with adaptive forgetting disabled and a real Ollama embedder forced, never the synthetic-hash fallback) for the apples-to-apples retrieval-capability number, AND the shipped as-designed config (MemOS tree_text; OpenMemory with its default forgetting) so the comparison is honest about both retrieval quality and each system's own design tradeoff.

KILL CRITERION (validity and fair-play, fixed before the run). This is a comparison, not a lever: the honest scoreboard is the finding whatever the order. A system's run is INVALID and re-run/flagged, not scored, if (a) more than 10% of generations are degenerate (the generator is our own qwen3.5:9b which we know handles /no_think, so this guards a setup error, not the model), or (b) a documented setup trap fired and silently collapsed recall (OpenMemory's synthetic-embedder fallback, or its forgetting fingerprinting the embeddings, or MemOS defaulting to the wrong text_mem backend), detected by a near-floor retrieval-hit rate on a sanity slice before the full run. FAIR-PLAY pre-commitments: identical generator, judge, data, and top-k for all three; the judge is frozen; we publish the FULL matrix (taOSmd, MemOS general_text + tree_text, OpenMemory forgetting-off + as-designed) regardless of who wins, with no dropping a config because a peer beat us or because we beat a peer; the forgetting-disabled and raw-store conditions are declared here, before any number, as the fair retrieval comparison, with the shipped-default numbers reported alongside and labelled. taOSmd's own number on this set is the F-013 reference recomputed under the identical harness, not the historical 74.6% quoted across configs. Integration risk is mitigated by a per-system sanity slice (a handful of questions, inspect that retrieval returns real on-topic memories) before each full run; open interface details from the recon (MemOS create_user/register_mem_cube signatures and the explicit general_text backend; OpenMemory's Python search() result keys) are resolved at adapter-build time, not assumed.
